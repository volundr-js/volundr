Migration from discord.js

A side-by-side mapping for developers coming from discord.js.

Installation

discord.jsVölundr
npm install discord.jsnpm install @volundr/client

Client setup

discord.jsVölundr
new Client({ intents: [GatewayIntentBits.Guilds] }) new Client({ token, intents: GatewayIntents.Guilds })
client.login(token) client.connect() (token is in constructor)
client.destroy() client.disconnect()

Intents

discord.jsVölundr
GatewayIntentBits.Guilds GatewayIntents.Guilds
[IntentA, IntentB] (array) IntentA | IntentB (bitwise OR)

Events

discord.jsVölundr
Events.MessageCreate / "messageCreate" "MESSAGE_CREATE" or GatewayEvent.MessageCreate
Events.InteractionCreate "INTERACTION_CREATE"
Events.ClientReady "ready"

All Discord events use native SCREAMING_SNAKE_CASE names. No translation layer.

Sending messages

discord.jsVölundr
channel.send("text") channel.send("text") (same)
channel.send({ embeds: [embed] }) channel.send(msg => msg.addEmbed(embed))
message.reply("text") message.reply("text") (same)
interaction.reply({ content: "hi", ephemeral: true }) interaction.reply(msg => { msg.content = "hi"; msg.ephemeral = true; })

Cache

discord.jsVölundr
client.guilds.cache client.guilds (already a Collection)
guild.members.cache guild.members
.cache.get(id) .get(id)
.cache.find(fn) .find(fn)

Fetching

discord.jsVölundr
guild.members.fetch(id) guild.members.fetch(id) or client.fetchGuildMember(guildId, userId)
channel.messages.fetch(id) client.fetchMessage(channelId, messageId)

In Völundr: .get() reads from cache, .fetch() calls the API. Always.

Collections

Völundr's Collection is almost identical to discord.js's. Same .filter(), .map(), .find(), .first(), .last(), .random(), etc.

Interactions

discord.jsVolundr
if (interaction.commandName === "ping") client.command("ping", handler)
if (interaction.customId === "btn") client.button("btn", handler)
if (interaction.isModalSubmit()) client.modal("id", handler)

Declarative routing replaces the if/switch pattern. You can still use client.on("INTERACTION_CREATE") if you prefer.

Key differences

  • Token in constructor - no separate .login() call.
  • No .cache layer - client.guilds is the collection.
  • Native event names - MESSAGE_CREATE not messageCreate.
  • Declarative routing - client.command(), client.button() instead of manual if/switch.
  • Tagged templates - fmt`Hello ${user}!` auto-resolves mentions.
  • Event streaming - client.stream("EVENT") returns an AsyncGenerator.
  • Resource management - await using client = new Client(...) via Symbol.asyncDispose.
  • Callback builder pattern - msg => { msg.content = "..." } in addition to options objects.
  • Bitwise intents - A | B not [A, B].
  • No managers - direct methods on Client instead of guild.channels.create().
