Völundr

A Discord SDK for TypeScript. Fast, predictable, zero magic.

Node.js 18+  •  Discord API v10  •  TypeScript 5+

Why Völundr?

Predictable API. .get() reads from cache. .fetch() calls the API. .send() accepts the same input everywhere. You can guess the method before opening docs.

No abstraction hell. Three layers: gateway, REST, entities. No 20 intermediate managers. You always know what's happening.

Fast where it matters. 500k+ events/sec. Sub-microsecond latency. 2.4× less GC garbage than Node.js EventEmitter. But you'll never need to think about this - it just doesn't lag.

Scales. Cache sweepers, configurable limits, worker thread sharding, global rate limit coordination. Built for bots that run for months on hundreds of shards.

Mental Model

Discord API
     │
  Gateway ──→ events (WebSocket, shards, compression)
  REST    ──→ requests (rate limits, retries, buckets)
     │
  Client  ──→ entities + cache + builders
     │
  Your bot

Three layers. Gateway handles WebSocket events. REST handles HTTP requests. Client ties them together with rich entities, cache, and builder DSL.

What makes it different

Not a discord.js clone. Different design decisions from the ground up.

  • Declarative routing. client.command("ping", handler) and client.button(/^ticket-\d+$/, handler) instead of manual if/switch on interaction names and custom IDs.
  • Modern resource management. await using client = new Client(...) - automatic cleanup via Symbol.asyncDispose. Collectors support Symbol.dispose.
  • Event streaming. for await (const msg of client.stream("MESSAGE_CREATE")) - consume events as async iterables with AbortSignal cancellation.
  • Tagged template mentions. fmt`Hello ${user}!` auto-resolves entities to Discord mention syntax.
  • Integrated builders. Functional helpers (embed(), row(), button()) and callback DSL (msg => { msg.embed(...) }) ship with the client - no separate package needed.
  • Built-in thread pool. ThreadPool for multi-core shard processing using worker threads, no external sharder process.
  • TypeScript-first. Type predicates (interaction.isChatInputCommand()), typed events, generics throughout. Not types bolted onto JavaScript.
  • Single import. @volundr/client includes everything - entities, builders, cache, collectors, permissions, formatters. Most bots need one dependency.

21,000+ lines of source. 486 tests. 130+ REST methods. 8 packages. Zero external runtime dependencies.

Quick Start

From zero to a working bot in under a minute.

Install

npm install @volundr/client

The simplest bot

import { Client } from "@volundr/client";
import { GatewayIntents } from "@volundr/gateway";

const client = new Client({
    token: process.env.DISCORD_TOKEN!,
    intents: GatewayIntents.Guilds,
});

client.command("ping", i => i.reply("Pong!"));
client.connect();

4 lines of actual logic. No boilerplate, no if/switch, no 47 imports.

A real bot in 30 lines

import { Client, fmt, embed, row, button } from "@volundr/client";
import { GatewayIntents } from "@volundr/gateway";
import { ButtonStyle } from "@volundr/types";

const client = new Client({
    token: process.env.DISCORD_TOKEN!,
    intents: GatewayIntents.Guilds,
});

// Slash command with embed + button
client.command("userinfo", async (i) => {
    const user = i.options.getUser("target", i.client) ?? i.user;
    await i.reply({
        embeds: [embed({ title: user.displayName, thumbnail: user.displayAvatarURL(), color: 0x5865F2 })],
        components: [row(button("refresh", "Refresh", ButtonStyle.Primary))],
    });
});

// Button handler with regex matching
client.button("refresh", async (i) => {
    await i.update(fmt`Refreshed by ${i.user}!`);
});

// Autocomplete
client.autocomplete("search", async (i) => {
    const query = i.focused?.value ?? "";
    await i.respond([{ name: `Result: ${query}`, value: String(query) }]);
});

client.connect();

Commands, buttons, autocomplete, embeds, tagged templates - all in 30 lines. No frameworks, no file-based routing, no decorators. Just TypeScript.

Tip: Every .command(), .button(), .selectMenu(), and .modal() returns this, so you can chain them. RegExp matching on custom_id lets you route ticket-123, ticket-456 etc. with a single handler.

Full Example

A complete bot with slash commands, buttons, embeds, and event handling.

Project setup

mkdir my-bot && cd my-bot
npm init -y
npm install @volundr/client
npm install -D tsx typescript

Create a .env file:

DISCORD_TOKEN=your-token-here

Full bot code

Save this as src/index.ts:

import "dotenv/config";
import { Client, fmt, embed, row, button, SlashCommandBuilder } from "@volundr/client";
import { GatewayIntents, ButtonStyle } from "@volundr/types";

const client = new Client({
    token: process.env.DISCORD_TOKEN!,
    intents: GatewayIntents.Guilds
           | GatewayIntents.GuildMessages
           | GatewayIntents.MessageContent,
});

// Register slash commands on ready
client.once("ready", async () => {
    console.log(`Logged in as ${client.user!.tag}`);

    await client.setGlobalCommands([
        new SlashCommandBuilder()
            .setName("ping")
            .setDescription("Check bot latency")
            .toJSON(),
        new SlashCommandBuilder()
            .setName("userinfo")
            .setDescription("Show info about a user")
            .addUserOption(opt =>
                opt.setName("user").setDescription("Target user")
            )
            .toJSON(),
    ]);
});

// Slash commands - declarative routing
client.command("ping", async (interaction) => {
    const start = Date.now();
    await interaction.reply("Pinging...");
    const latency = Date.now() - start;

    await interaction.editReply({
        embeds: [embed({
            title: "Pong!",
            description: `Latency: **${latency}ms**`,
            color: 0x00ff88,
        })],
    });
});

client.command("userinfo", async (interaction) => {
    const target = interaction.options.getUser("user", interaction.client) ?? interaction.user;

    await interaction.reply({
        embeds: [embed({
            title: target.displayName,
            thumbnail: target.displayAvatarURL(),
            fields: [
                { name: "ID", value: target.id, inline: true },
                { name: "Created", value: target.createdAt.toDateString(), inline: true },
            ],
            color: 0x5865F2,
        })],
        components: [row(
            button("avatar-link", "Avatar URL", ButtonStyle.Link, { url: target.displayAvatarURL({ size: 1024 }) }),
        )],
    });
});

// Message listener
client.on("MESSAGE_CREATE", async (message) => {
    if (message.author.bot) return;

    if (message.content === "!hello") {
        await message.reply(fmt`Hey ${message.author}!`);
    }
});

// Connect
client.connect();

Run it

npx tsx src/index.ts

Events

Listen to Discord events with type-safe handlers.

Listening to events

client.on("MESSAGE_CREATE", (message) => {
    console.log(message.content);
});

client.on("GUILD_MEMBER_ADD", (member) => {
    console.log(`${member.user.username} joined ${member.guild.name}`);
});

client.on("ready", () => {
    console.log("Bot is online!");
});

Event names

Völundr uses Discord's native event names in SCREAMING_SNAKE_CASE. No translation layer. You can use string literals or the GatewayEvent enum:

import { GatewayEvent } from "@volundr/types";

// Both are equivalent
client.on("MESSAGE_CREATE", (message) => { /* ... */ });
client.on(GatewayEvent.MessageCreate, (message) => { /* ... */ });

Rich entities

5 events emit rich entity objects with methods. All others emit raw Discord API data.

EventPayloadDescription
MESSAGE_CREATEMessageRich entity with .reply(), .delete(), .react()
MESSAGE_UPDATEMessageRich entity (only if cached)
INTERACTION_CREATEInteractionRich entity with .reply(), .deferReply(), etc.
GUILD_CREATEGuildRich entity (only if cached)
GUILD_MEMBER_ADDGuildMemberRich entity (only if cached)
 
client.on("MESSAGE_CREATE", (message) => {
    // message is a Message entity, not raw API data
    message.author.username;    // User entity
    message.guild;              // Guild entity (or null in DMs)
    message.channel;            // Channel entity
    message.reply("Hello!");    // method - calls REST API
});

Once

Listen for an event only once:

client.once("ready", () => {
    console.log("First ready!");
});

Event streaming

Consume events as an async iterable with client.stream(). Cancel with an AbortSignal:

// Process messages as a stream
const ac = new AbortController();

for await (const message of client.stream("MESSAGE_CREATE", { signal: ac.signal })) {
    console.log(message.content);
    if (message.content === "stop") ac.abort();
}

Useful for sequential workflows where you need to wait for a specific event:

// Wait for a specific user's message
async function waitForResponse(channelId: string, userId: string): Promise {
    for await (const msg of client.stream("MESSAGE_CREATE")) {
        if (msg.channel?.id === channelId && msg.author.id === userId) {
            return msg.content;
        }
    }
    return "";
}

Sending Messages

Three ways to send. Same result. Works everywhere.

The golden rule

Every .send() and .reply() in Völundr accepts the same three input types:

1. String

channel.send("Hello!");

2. Options object

channel.send({
    content: "Hello!",
    embeds: [embed.toJSON()],
});

3. Callback builder

channel.send(msg => {
    msg.content = "Hello!";
    msg.embed(e => {
        e.title = "Welcome";
        e.description = "This is an embed";
        e.color = 0x5865F2;
        e.field("Users", "1,234", true);
        e.field("Servers", "56", true);
    });
    msg.row(
        new ButtonBuilder().setCustomId("ok").setLabel("Got it").setStyle(ButtonStyle.Primary),
    );
});
Tip: The callback builder requires zero imports. Build embeds, buttons, and action rows inline.

Works on every entity

The same MessageInput type is accepted by:

  • TextChannel.send()
  • DMChannel.send()
  • VoiceChannel.send()
  • ThreadChannel.send()
  • User.send()
  • Message.reply()

Slash Commands

Register and handle application commands.

Declarative routing

The recommended way. Register handlers with client.command() and let Volundr route interactions automatically:

import { SlashCommandBuilder } from "@volundr/client";

// Register the command with Discord
const ping = new SlashCommandBuilder()
    .setName("ping")
    .setDescription("Replies with pong!")
    .toJSON();

await client.setGlobalCommands([ping]);

// Handle it
client.command("ping", (interaction) => {
    interaction.reply("Pong!");
});

Command with options

const greet = new SlashCommandBuilder()
    .setName("greet")
    .setDescription("Greet someone")
    .addUserOption(opt =>
        opt.setName("user").setDescription("Who to greet").setRequired(true)
    )
    .addStringOption(opt =>
        opt.setName("message").setDescription("Custom greeting")
    )
    .toJSON();

client.command("greet", (interaction) => {
    const user = interaction.options.getUser("user", interaction.client);
    const msg = interaction.options.getString("message") ?? "Hello!";
    interaction.reply(fmt`${msg} ${user}!`);
});

Subcommands

Use "name/sub" or "name/group/sub" syntax to route directly to subcommands:

const config = new SlashCommandBuilder()
    .setName("config")
    .setDescription("Server configuration")
    .addSubcommandGroup(group =>
        group.setName("channel").setDescription("Channel settings")
            .addSubcommand(sub =>
                sub.setName("set").setDescription("Set a channel")
                    .addChannelOption(opt =>
                        opt.setName("channel").setDescription("The channel").setRequired(true)
                    ),
            )
            .addSubcommand(sub =>
                sub.setName("remove").setDescription("Remove channel setting")
            ),
    )
    .toJSON();

// Route directly to subcommands
client.command("config/channel/set", (interaction) => {
    const ch = interaction.options.getChannel("channel", interaction.client);
    interaction.reply(fmt`Config channel set to ${ch}`);
});

client.command("config/channel/remove", (interaction) => {
    interaction.reply("Config channel removed.");
});
Fallback: If no subcommand handler matches, Volundr falls back to the base command handler (client.command("config", ...)). This lets you handle all subcommands in one place if you prefer.

Collectors

Collect messages, button clicks, and reactions over time.

Message collector

const collector = channel.createMessageCollector({
    filter: (msg) => msg.author.id === userId,
    time: 30_000,
    max: 5,
});

collector.on("collect", (msg) => {
    console.log(msg.content);
});

collector.on("end", (collected) => {
    console.log(`Got ${collected.size} messages`);
});

Component collector

const collector = message.createComponentCollector({
    filter: (i) => i.user.id === userId,
    time: 60_000,
});

collector.on("collect", (interaction) => {
    if (interaction.customId === "next") {
        interaction.update({ content: "Page 2" });
    }
});

Await shorthand

// Wait for one button click (30s timeout)
const clicks = await message.awaitComponents({
    time: 30_000,
    max: 1,
});

Reaction collector

const collector = message.createReactionCollector({
    filter: (reaction, user) => reaction.emoji.name === "👍",
    time: 60_000,
});

collector.on("collect", (reaction, user) => {
    console.log(`${user.username} reacted!`);
});

Automatic cleanup with using

Collectors implement Symbol.dispose, so you can use the using keyword for automatic cleanup:

// Collector is automatically stopped when it goes out of scope
{
    using collector = channel.createMessageCollector({
        filter: (msg) => msg.author.id === userId,
        time: 30_000,
    });

    collector.on("collect", (msg) => {

Error Handling

How to handle API errors, rate limits, and disconnects.

REST errors

All REST calls can throw DiscordAPIError or HTTPError.

import { DiscordAPIError, HTTPError } from "@volundr/rest";

try {
    await message.reply("Hello!");
} catch (err) {
    if (err instanceof DiscordAPIError) {
        console.log(err.code);       // Discord error code (e.g. 50013)
        console.log(err.message);    // "Missing Permissions"
        console.log(err.status);     // HTTP status (e.g. 403)
        console.log(err.method);     // "POST"
        console.log(err.url);        // "/channels/123/messages"
    }

    if (err instanceof HTTPError) {
        console.log(err.status);     // 500, 502, etc.
    }
}

Rate limits

Völundr handles rate limits automatically. The REST client queues requests per route and respects Retry-After headers. You don't need to do anything - it just works.

// These are safe - the client queues them automatically
await Promise.all([
    channel.send("Message 1"),
    channel.send("Message 2"),
    channel.send("Message 3"),
]);

If you hit a global rate limit, all requests are paused until it expires. The REST client logs a warning when this happens.

Gateway reconnects

The gateway automatically reconnects on disconnect. It resumes the session when possible (keeping events), and starts a new session when not. This is all handled internally.

// You can listen for lifecycle events
client.on("ready", () => {
    console.log("Connected!");
});

// The gateway emits debug info you can log
client.gateway?.on("debug", (msg) => {
    console.log("[GW]", msg);
});

Unhandled rejections

Always handle async errors in event listeners. Unhandled promise rejections will crash your bot in Node.js 22+.

// Bad - unhandled rejection if reply fails
client.on("MESSAGE_CREATE", (message) => {
    message.reply("Hi"); // no await, no catch
});

// Good - catch errors
client.on("MESSAGE_CREATE", async (message) => {
    try {
        await message.reply("Hi");
    } catch (err) {
        console.error("Failed to reply:", err);
    }
});
Tip: A global safety net: process.on("unhandledRejection", console.error). This prevents crashes but don't rely on it - handle errors at the source.

Caching

How Völundr manages data in memory.

How cache works

Völundr caches entities automatically as events arrive from the gateway. The cache is the single source of truth for your bot.

The two rules

  • .get(id) - reads from cache. Instant, never hits the API.
  • .fetch(id) - calls the Discord API. Updates cache with the response.
// From cache (instant, may be null)
const guild = client.guilds.get("123456");

// From API (always fresh, updates cache)
const guild = await client.fetchGuild("123456");

Sweepers

Automatically clean up old cache data to control memory usage.

Configuration

import { Client, messageLifetimeFilter } from "@volundr/client";

const client = new Client({
    token: "...",
    intents: ...,
    sweepers: {
        messages: {
            interval: 60_000,           // run every 60 seconds
            filter: messageLifetimeFilter(30 * 60_000), // remove messages older than 30 min
        },
        presences: {
            interval: 300_000,
            filter: () => true,        // clear all presences every 5 min
        },
    },
});

Sharding

Scale your bot across multiple WebSocket connections.

Auto sharding

Völundr uses worker thread sharding - more memory-efficient than process-based sharding.

const client = new Client({
    token: "...",
    intents: ...,
    shardCount: "auto",  // Discord tells us how many shards
});

Manual shard count

const client = new Client({
    token: "...",
    intents: ...,
    shardCount: 4,
});

Thread pool

For fine-grained control over worker threads:

import { ThreadPool } from "@volundr/client";

const pool = new ThreadPool({
    workerPath: "./worker.js",

When to shard: Discord requires sharding at 2,500 guilds. Below that, a single connection is fine.

Why it's fast

Event pipeline

WebSocket → JSON.parse → _patch() → cache → emit

Zero intermediate copies. Entities are patched in-place - MESSAGE_UPDATE only touches changed fields.

Zero-allocation design

  • Entities reuse objects. _patch() mutates in place instead of creating new instances.
  • Lazy initialization - Date objects and Set collections are only created when first accessed.
  • 2.4× less GC garbage than Node.js EventEmitter under sustained load.

Memory efficiency

  • 259 bytes per cached member (vs 400+ in typical implementations).
  • Configurable sweepers prevent unbounded cache growth.
  • Worker thread sharding shares the V8 heap more efficiently than child processes.
